# 0020.1. Static Site Generation Build Toolkit

**Date**: 2026-04-19
**Status**: Proposed
**Depends on**: [0020-static-rendering-template-engine](./0020-static-rendering-template-engine.md)

## Context

ADR 0020 added `@nisli/core/static` as a DOM-free string renderer:

- `staticHtml`
- `renderToString`
- `raw`
- `StaticResult`
- `RawHtml`

That implementation is useful as a low-level primitive and replaced the
publication site's local static renderer. But it exposed two larger product
problems:

1. If SSG requires authors to replace `html` with `staticHtml` and `component()`
   with a new static component API, then Nisli has created a second component
   authoring model.
2. If `@nisli/core/static` remains the public SSG home, then the core UI runtime
   owns build tooling, content discovery, route output, feeds, and asset
   bundling.

That is the wrong direction.

The publication site already has components built with Nisli's normal browser
framework. SSG should make those components useful for SEO and build-time HTML.
Authors should not have to rebuild their component library against a separate
static API.

## Decision

The next phase is a dedicated **`@nisli/ssg` package**.

`@nisli/core` should return to being the browser/client UI framework:

- `component`
- `html`
- signals
- dependency injection
- query
- lifecycle
- refs
- browser runtime behavior

`@nisli/ssg` should own static and publication build-time tooling:

- `buildStaticSite`
- route-to-file output
- content pipeline hooks
- metadata/feed/sitemap outputs
- public asset copying
- client bundle and island coordination
- internal component-to-static serialization
- internal static HTML primitives such as `staticHtml`, `raw`, and
  `renderToString`

Because the framework currently has only one known consumer, there is no need to
carry a compatibility bridge. Remove `@nisli/core/static` and move the static
renderer to `@nisli/ssg`.

SSG is a build target, not a component authoring mode. Users should continue to
author components with the normal public API:

```ts
import { component, html } from '@nisli/core';

const PostCard = component<{ title: string; href: string }>('post-card', props => html`
  <article>
    <a href="${props.href}">${props.title}</a>
  </article>
`);
```

The public API should be the build pipeline, not a component renderer. A
publication should call a Nisli-provided build tool that owns route generation,
file output, metadata, assets, and static HTML generation:

```ts
import { buildStaticSite } from '@nisli/ssg';

await buildStaticSite({
  outDir: 'dist',
  context: { posts },
  routes: [
    { path: '/', render: ({ posts }) => HomePage({ posts }) },
    { path: '/about', render: () => AboutPage({}) },
  ],
});
```

The authoring model remains Nisli:

- `component()`
- `html`
- factory composition
- props as normal factory inputs
- signals inside component setup
- `class:name`
- `html:inner`
- nested component factories

Inside that build tool, Nisli may have an internal component renderer that
decides which parts can become static HTML, which parts are ignored as
browser-only behavior, and which components should remain custom-element
islands. That renderer is an implementation detail, not the primary customer API.

`staticHtml` and any internal component renderer remain useful implementation
primitives, but they should not be exported as normal customer APIs. The primary
SSG promise is:

> Write normal Nisli components. Render them at build time when they are
> statically renderable by running the Nisli SSG build.

## Corrected Mental Model

The previous idea of adding `staticComponent()` would have created a parallel
framework:

```ts
// Wrong direction for primary authoring:
import { staticComponent, staticHtml } from '@nisli/ssg';
```

That loses compatibility with components already written for `@nisli/core`.

The corrected model is:

- `@nisli/core` owns component authoring and the browser runtime
- `@nisli/ssg` owns publication build-time tooling
- SSG builds routes, files, metadata, assets, and existing component trees
- browser-only behavior either disappears from static HTML or becomes an island
- SSR can later reuse the same renderer, but request-time SSR is not the current
  product goal

The distinction between SSG and SSR is timing:

- **SSG** renders the component tree during the build and writes files.
- **SSR** renders the component tree during a request.

Both need a component renderer somewhere in the stack. For SSG, that renderer
should sit behind the build tool rather than become the customer-facing API.

## Cross-Reference Table

| Concern | Browser CSR | SSG Build Tool | Decision |
| --- | --- | --- | --- |
| Authoring import | `import { component, html } from '@nisli/core'` | Same component source should stay valid | Do not require `staticHtml` for primary component authoring. |
| Component definition | `component(tag, setup)` registers a custom element and returns a factory | Build tool should internally render static-safe factory results | Reuse existing `component()` API. |
| Template primitive | `html` returns a `TemplateResult` mounted into DOM | Build tool should internally serialize the same `TemplateResult` to HTML | Add internal static rendering support for existing `html` results. |
| Props | Factory props accept plain values or signals; setup receives `Signal<T>` props | Plain build-time props become one-shot setup signals | Preserve prop model. |
| Text interpolation | Signal values update DOM text nodes | Signal values are read once and escaped into HTML | Reuse signal read semantics without subscriptions. |
| `computed()` | Lazily tracks dependencies and updates when inputs change | Can be evaluated once during render | Supported for pure synchronous derived values. |
| `effect()` | Side effects with component-scoped disposal | Should not run for static HTML by default | Browser-only unless a future explicit server effect model exists. |
| `class:name` | Updates `classList` reactively | Resolve once into the serialized `class` attribute | Supported as static class evaluation. |
| Attribute bindings | Set attributes/properties and update on signal changes | Resolve once into escaped attributes | Supported; needs careful static attribute serializer. |
| Event handlers | `@click`, modifiers, and handler wrapping | No static HTML representation | Omit from static output; interactive behavior requires island bundle. |
| Refs | Receive real DOM elements | No DOM exists | Unsupported in SSG render. |
| Lifecycle | `onMount`, `onCleanup`, connect/disconnect | No browser mount occurs | Unsupported in SSG render; islands run lifecycle in browser. |
| DI | `inject()` / `provide()` use component setup context | Could work for synchronous, build-safe services later | Defer unless the publication use case requires it. |
| `query()` | Reactive async data loading/cache | Publication data should be loaded before render | Defer; SSG route loaders are a higher-level concern. |
| `html:inner` | Trusted DOM inner HTML | Trusted serialized HTML | Supported with explicit trust boundary. |
| Nested component factories | Mount child custom elements and forward props/signals | Build tool renders child factories recursively when static-safe | Required for useful SSG. |
| Islands | Browser runtime upgrades custom elements | Build tool emits custom-element tags plus required scripts for interactive components | Required as the progressive enhancement boundary. |
| `staticHtml` | Not part of core | Internal low-level string template primitive in `@nisli/ssg` | Keep behind the build tool, not in the public barrel. |
| Routes/files | Application concern | Build writes `dist/<route>/index.html`, feeds, sitemap, assets | This is the public SSG abstraction. |
| Component renderer | Browser runtime mounts to DOM | Internal SSG implementation serializes static-safe component trees | Do not expose as the main customer API. |

## Renderability Rules

The SSG build tool should define a conservative static subset for the internal
component renderer. A component is statically renderable when its setup can run
synchronously without browser-only dependencies and returns an `html` template
that can be serialized.

Supported in the first useful version:

- component factories returned by `component()`
- plain factory props
- one-time signal props created for setup
- synchronous `computed()` reads
- nested `html` templates
- nested static-safe component factories
- text interpolation with escaping
- attribute interpolation with escaping
- `class:name` resolved once
- arrays and conditional children
- trusted HTML through the existing `html:inner` boundary

Browser-only during SSG:

- event listeners
- refs
- `onMount`
- DOM access through `host`
- imperative `host` mutation
- effects with side effects
- async query loading
- request-scoped services

When a component is not statically renderable but should remain interactive, the
SSG renderer should be able to emit it as an island:

```html
<nisli-theme-toggle></nisli-theme-toggle>
<script type="module" src="/assets/theme-toggle.js"></script>
```

The exact island declaration API is separate work, but the boundary is part of
this phase: static rendering generates initial HTML; the browser runtime upgrades
interactive components later.

## Problems To Solve Now

1. Package the publication build pipeline.

   The blog currently owns a custom pipeline that cleans `dist`, discovers
   Markdown and TypeScript posts, validates frontmatter, renders pages, writes
   route files, generates Open Graph images, RSS, sitemap, `llms.txt`,
   `llms-full.txt`, `posts.json`, copies assets, and bundles client JavaScript.
   Nisli should provide this shape out of the box so publications do not rebuild
   the same SSG scaffolding.

2. Expose enough component metadata for the internal renderer.

   Today `component()` hides the setup function behind custom-element
   registration and factory results. The SSG renderer needs a supported way to
   identify factory results, recover the tag name, props, host attrs, and render
   function, and do so without relying on accidental private object shapes.

3. Serialize `html` `TemplateResult` without a browser DOM.

   Browser `html` currently renders by inserting markers, parsing with
   `document.createElement('template')`, walking nodes, and wiring bindings. SSG
   needs a parallel serializer that understands the same template directives but
   produces strings.

4. Preserve the factory composition model.

   Existing component call sites such as `PostCard({ post })` should work in CSR
   and SSG. Authors should not switch to `staticComponent()` or rewrite
   templates with `staticHtml`.

5. Define static semantics for reactive primitives.

   Signals and computed values are useful at build time as one-shot values. They
   are not live subscriptions in SSG output. Effects and lifecycle are
   browser-only until an explicit server lifecycle exists.

6. Keep islands honest.

   Interactive components are still normal Nisli components. SSG should either
   render their static shell when safe or emit their custom-element tag for
   browser upgrade. It should not pretend event handlers work in static HTML.

## Proposed Public API Direction

Primary SSG API:

```ts
import { buildStaticSite } from '@nisli/ssg';

await buildStaticSite({
  outDir: 'dist',
  context: { posts },
  routes: [
    { path: '/', render: ({ posts }) => HomePage({ posts }) },
    ...posts.map(post => ({
      path: `/posts/${post.slug}`,
      render: () => PostPage({ post }),
    })),
  ],
  assets: {
    publicDir: 'public',
    clientEntries: ['src/client.ts'],
  },
  outputs: {
    feed: true,
    sitemap: true,
    llms: true,
    postsJson: true,
  },
});
```

Internal low-level implementation APIs:

```ts
staticHtml`...`;
raw(trustedHtml);
renderToString(value);
```

These are not the preferred component authoring surface and should not be
exported from the public package barrel. They are implementation primitives for
trusted pre-rendered fragments, feeds, and component serialization inside
`buildStaticSite()`.

Internal-only implementation API:

```ts
renderComponentToStaticHtml(App({ posts }));
```

The name and shape do not matter because this should not be the normal exported
customer API. It exists so the build tool can turn existing Nisli component
trees into HTML.

APIs to avoid as primary direction:

```ts
staticComponent(...)
staticHtml`...`
```

Those create a second component model and should not be required for SSG.
`staticHtml` can exist internally, but customers should not need to import it.

## Publication Use Case

The immediate target is a personal web publication built with Nisli components.
The site needs:

- SEO-friendly static HTML
- reuse of existing `@nisli/core` components
- Markdown body insertion through an explicit trusted boundary
- typed TypeScript article modules
- reusable editorial blocks written as normal Nisli components where possible
- metadata, feeds, sitemap, and `llms.txt` from Nisli's SSG build pipeline
- custom-element islands for small interactive behavior
- no request-time rendering requirement

The next mandatory framework capabilities are:

1. `buildStaticSite()` or equivalent publication build API
2. content discovery for Markdown and TypeScript article modules
3. route-to-file output such as `dist/<slug>/index.html`
4. metadata output for canonical tags, Open Graph, RSS, sitemap, `llms.txt`, and
   `posts.json`
5. client asset bundling and island script coordination
6. internal static serialization for existing `html` templates
7. internal nested component factory rendering
8. docs explaining SSG as a build target, not a new authoring mode

## Non-Goals

This phase does not include:

- a new `staticComponent()` authoring model
- a public `renderStatic()`-style component renderer as the primary customer API
- request-time SSR server APIs
- streaming
- hydration
- DOM adoption
- running browser event handlers at build time
- running `onMount` at build time
- giving refs fake DOM nodes
- async query orchestration inside component setup

Routes, Markdown, feeds, sitemap, and dev-server behavior are no longer deferred
as a separate future concern. They are the SSG product surface. The internal
component renderer supports that build tool, but users should not need to call it
directly.

## Consequences

Positive:

- Existing Nisli component libraries remain valuable for SSG.
- Authors do not learn a second component model.
- The framework's factory-first philosophy stays intact.
- SSG output improves SEO without changing component source.
- The blog's custom build pipeline can move into Nisli-owned tooling over time.
- Future SSR can reuse the component renderer while keeping request-time server
  APIs separate.

Negative:

- This is harder than exposing `staticHtml`; it requires both SSG pipeline
  tooling and a real serializer for existing `html` and component factory
  results.
- Some browser components will not be statically renderable without refactoring
  browser-only behavior behind islands.
- The component/runtime internals need supported metadata for the SSG renderer.

Mitigations:

- Start with a conservative renderable subset.
- Make unsupported browser-only features fail clearly or become explicit islands.
- Keep route/build-system concerns in the public SSG layer and component
  serialization behind it.
- Keep `staticHtml` as an internal low-level primitive, but do not export it as
  the SSG authoring path.

The next phase should make `@nisli/ssg` useful enough for the publication site
without freezing every blog-specific policy into the framework.

parts of `/Users/goga/Documents/goga/blog/packages/blog/src/pipeline/build.ts`
without forcing the blog to give up its content model.

Move into `@nisli/ssg` or use from `@nisli/ssg`:

- route-to-file writing
- root file writing
- public asset copying
- page build hooks
- build result reporting

Keep in the blog for now:

- section taxonomy
- page shell design
- JSON-LD shape
- RSS copy and feed policy
- sitemap URL policy
- `llms.txt` formatting
- OG image design
- Turndown rules for publication-specific custom elements

Success criteria:

- the blog can depend on `@nisli/ssg` instead of its own generic output helpers
- `pnpm -C packages/blog typecheck` passes
- `pnpm -C packages/blog build` passes
- generated output is equivalent for representative pages and root files

### Phase 2B: Add Publication Primitives Behind Hooks

Goal: reduce repeated publication plumbing while avoiding a hard-coded blog
engine.

Candidate additions:

- `writeRoute()` / `writeRoot()` public helpers if `buildStaticSite()` needs
  lower-level escape hatches
- route manifest expansion helpers for arrays of posts
- canonical URL helpers
- sitemap helper
- RSS helper
- JSON output helper for `posts.json`
- `llms.txt` helper only if the shape proves reusable
- typed build context helpers

These should be hook-based and data-driven. They should not assume the blog's
exact frontmatter fields unless those fields are passed through typed generic
configuration.

### Phase 2C: Component-Aware Internal Rendering

Goal: make SSG render normal Nisli components without exposing a public
`renderStatic()` API.

Required framework work:

- expose stable internal metadata for component factory results
- preserve the public `component()` / `html` authoring model
- serialize existing `html` `TemplateResult` values without a browser DOM
- evaluate signal and computed values once during build
- serialize text, attributes, `class:name`, `html:inner`, arrays, conditionals,
  and nested component factories
- omit or reject browser-only event handlers, refs, lifecycle hooks, effects, and
  imperative `host` mutation
- return explicit island records for components that should upgrade in the
  browser

Customer-facing API should remain:

```ts
await buildStaticSite({
  routes: [
    { path: '/', render: ({ posts }) => HomePage({ posts }) },
  ],
});
```

The component serializer can exist internally, but it should not become the
normal API users import.

### Phase 2D: Island Asset Coordination

Goal: let the SSG build know which client bundles are needed by emitted custom
element islands.

Candidate shape:

- route-level island declarations
- component metadata that declares a client entry
- build result containing required assets per page
- script injection hook controlled by the app shell

This remains progressive enhancement, not hydration. Static HTML should be useful
before the browser bundle runs.

### Phase 2E: Publishing Hardening

Goal: make releases boring for both packages.

Tasks:

- configure npm Trusted Publisher for `@nisli/ssg`
- confirm the workflow skips already-published package versions
- use package-specific tags: `core-vX.Y.Z`, `ssg-vX.Y.Z`
- delete failed `ssg-v0.1.0` release/tag only if retrying the same unpublished
  version through CI
- keep `@nisli/core` and `@nisli/ssg` versions independent
- consider adding a package-scoped changelog once both packages evolve regularly
